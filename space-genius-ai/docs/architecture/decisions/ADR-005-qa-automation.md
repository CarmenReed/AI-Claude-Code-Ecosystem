# ADR-005: QA Automation and AI-Generated Test Architecture

**Status:** Accepted  
**Date:** 2026-04-12  
**Deciders:** Carmen Reed  
**Pain point addressed:** Low Playwright test coverage, manual QA scope definition

---

## Context

SpaceGenius has Gherkin acceptance criteria in ADO PBIs (generated by the documentation pipeline in ADR-001) but no automated mechanism for converting those ACs into Playwright test files. Test coverage grows only when a developer manually writes tests. QA scope is defined by what developers choose to document in testing notes.

The solution requires a closed-loop system: ADO PBI Gherkin ACs in, Playwright test file out, test results back to the PBI. The constraint is that Copilot Enterprise is not licensed, which is the natural tool for this kind of repo-context-aware test generation.

---

## Decision

Implement a Claude Code slash command `/generate-tests` that:
1. Reads Gherkin ACs from the linked ADO PBI via ADO REST API (PAT from Key Vault MCP)
2. Reads relevant source code files using full CLAUDE.md codebase context
3. Generates a Playwright test file with one test per AC scenario, including setup, actions, and assertions
4. Commits the test file to the feature branch
5. Existing ADO Pipelines runs Playwright on the next CI trigger
6. Pass/fail results are posted back to the PBI as a structured comment

The developer reviews the generated test file before committing. Human review is required.

---

## Alternatives Considered

| Option | Pros | Cons |
|---|---|---|
| Claude Code + CLAUDE.md + ADO REST (chosen) | No new Azure resources, closed loop from PBI to test to result, uses existing Pipelines infrastructure | CLAUDE.md requires maintenance; slash command requires each developer's Claude Code configuration; context quality degrades if CLAUDE.md is outdated |
| GitHub Copilot Enterprise | Whole-repo context without CLAUDE.md maintenance, always current | Not licensed; requires enterprise agreement |
| Copilot Workspace | PBI-to-PR automation including test generation with one click | Requires Copilot Enterprise |
| Manual test authoring with AI chat assist | Developers already familiar with this pattern | Does not close the loop to ADO; no automatic test result posting; depends on developer initiative |
| Azure AI Foundry + custom test generation prompt flow | Org-wide shared prompt, versioned | AI Foundry not provisioned; same output as the slash command approach with more infrastructure |

---

## Consequences

**Positive:**
- Every PBI with Gherkin ACs can generate a Playwright test file on demand
- Test coverage grows proportionally with the backlog rather than with developer initiative
- The closed loop (PBI AC to test file to pass/fail to PBI comment) gives Jessica clear QA scope
- Zero new Azure resources required

**Negative:**
- CLAUDE.md requires active maintenance to keep codebase context current
- Each developer must configure Claude Code with the Key Vault MCP server
- Generated test quality depends on CLAUDE.md accuracy and Gherkin AC quality

**Dependency:** This solution depends on Gherkin ACs being present in PBIs. The documentation pipeline (ADR-001) is the upstream provider. If ADR-001 is not implemented, ACs must be authored manually.

---

## Azure Migration Path

1. Provision GitHub Copilot Enterprise: the CLAUDE.md maintenance burden is eliminated; all developers have whole-repo context automatically
2. Copilot Workspace turns ADO PBI into plan into code into PR including test files, with no slash command required
3. The manual `/generate-tests` invocation becomes automatic: Copilot Workspace triggers on PBI state change from design to ready
4. The ADO Pipelines run and result-posting workflow is unchanged; only the test file authoring is automated
