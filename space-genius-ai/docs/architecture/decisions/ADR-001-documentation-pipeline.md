# ADR-001: Documentation Pipeline Architecture

**Status:** Accepted  
**Date:** 2026-04-12  
**Deciders:** Carmen Reed  
**Pain point addressed:** Meeting notes to BRD/FRD/LOE/Gherkin ACs pipeline

---

## Context

SpaceGenius consistently loses business requirements in the gap between meetings and documentation. The team meets, talks through requirements, and generates informal notes. Those notes rarely become complete BRDs, FRDs, or Gherkin acceptance criteria in ADO before development begins. This is not a process failure. It is a cognitive load failure: the documentation work is manual, repetitive, and disconnected from the meeting where the knowledge was fresh.

The solution requires an LLM to convert unstructured meeting transcripts into structured documentation and create the corresponding ADO work items. The constraint is that SpaceGenius has no Azure OpenAI deployment, no AI Foundry, and no Logic Apps.

---

## Decision

Implement an Azure Function (HTTP trigger) that accepts a meeting transcript, calls the Claude API (key stored in Key Vault, retrieved via Managed Identity), generates structured JSON output (BRD, FRD, LOE template, Gherkin ACs), and calls the ADO REST API to create a PBI in design state with documents attached as ADO wiki pages.

Human review is required before the PBI advances from design to ready. The AI does not auto-promote work items.

---

## Alternatives Considered

| Option | Pros | Cons |
|---|---|---|
| Azure AI Foundry prompt flow + Logic Apps + ADO native connector | Team-shared versioned prompts, native ADO connector, evaluation metrics, no custom REST code | AI Foundry and Logic Apps not provisioned; requires executive approval for new resources |
| Power Automate + Azure OpenAI | Visual designer, native Office 365 integration | Azure OpenAI not deployed; Power Automate license requirements unclear |
| Custom Azure Function + Claude API (chosen) | Uses only provisioned resources, C#/.NET is Carmen's native stack, zero new provisioning | No org-wide prompt versioning, no visual designer for non-developers, no built-in eval metrics |
| Manual documentation with AI chat assist | Zero infrastructure | Does not solve the pipeline problem; still requires manual copy-paste and ADO entry |

---

## Consequences

**Positive:**
- Design state with complete documentation is available within minutes of a meeting
- Uses only currently provisioned Azure resources with no new cost
- Gherkin ACs generated by the pipeline feed directly into the QA automation solution (ADR-005)
- Black-box deficiency prompting prevents hallucinated requirements

**Negative:**
- Prompt quality depends on individual prompt engineering, not versioned org-wide templates
- Kathy and non-developers cannot modify the workflow without a developer deployment
- No built-in quality evaluation metrics; output quality must be verified by human review

---

## Azure Migration Path

1. Provision Azure AI Foundry with a GPT-4o deployment (or reuse existing Azure OpenAI if deployed for another initiative)
2. Author the documentation pipeline as an AI Foundry prompt flow with evaluation metrics
3. Replace the Azure Function with a Logic Apps workflow using the native ADO connector
4. Non-developers can modify prompts in AI Foundry without code deployments
5. Add a prompt evaluation step that scores Gherkin AC completeness against a defined rubric
6. Run history in Logic Apps provides a compliance-grade audit trail of all documentation generated
